# Практическое занятие №3: Настройки запуска и базовая защита веб-службы

## Цель
Реализовать веб-службу с упорядоченной загрузкой конфигурации из трёх источников, ранней валидацией настроек, ограничением доступа по CORS, защитой от превышения частоты запросов и двумя защитными заголовками. Добавить два режима работы (учебный и боевой), переключаемых только настройкой.

## Используемые технологии
- .NET 10 Minimal API
- Встроенный middleware CORS
- Rate Limiting (фиксированное окно)
- Защитные HTTP‑заголовки

## Функциональные возможности

### 1. Источники конфигурации и приоритет
Конфигурация загружается в следующем порядке (каждый следующий переопределяет предыдущий):
1. Файл `appsettings.json`
2. Переменные окружения
3. Аргументы командной строки

Таким образом, аргумент `--Mode=Production` имеет наивысший приоритет и переопределяет настройку, заданную в файлах или переменных окружения.

### 2. Ранняя валидация критичных параметров
После объединения настроек выполняются проверки:
- Список доверенных источников (`AllowedOrigins`) не пуст.
- Каждый URL является абсолютным и использует HTTPS.
- Лимиты запросов (`ReadPermitLimit`, `WritePermitLimit`, `WindowInSeconds`) строго положительны.
- Параметр `Mode` может принимать только `Development` или `Production`.

При любой ошибке приложение немедленно завершается с кодом 1 и выводит понятные сообщения. Ранняя остановка гарантирует, что некорректно сконфигурированный экземпляр не начнёт принимать запросы.

### 3. Защита CORS
Политика CORS разрешает запросы только с доменов, указанных в `AllowedOrigins`. Запросы с недоверенных источников (например, `Origin: https://evil.com`) обрабатываются сервером, но браузер не получит заголовок `Access-Control-Allow-Origin`, что блокирует доступ к данным для вредоносных сайтов.

### 4. Ограничение частоты запросов (Rate Limiting)
Настроены два лимита:
- **Read-маршруты** (GET `/api/items`) – по умолчанию до 30 запросов в минуту.
- **Write-маршруты** (POST `/api/items`) – до 5 запросов в минуту.

При превышении возвращается код `429 Too Many Requests`. В режиме `Development` тело ответа содержит пояснение, в `Production` – только заголовок `Retry-After`.

### 5. Защитные заголовки
Каждый ответ добавляет:
- `X-Content-Type-Options: nosniff` – предотвращает угадывание MIME-типов.
- `X-Frame-Options: DENY` – запрещает встраивание страницы во фреймы, защищая от кликджекинга.

### 6. Режимы работы
Учебный (`Development`) и боевой (`Production`) режимы переключаются через параметр `Mode` в любом из источников конфигурации.
- Development – подробные сообщения об ошибках, включая текст при превышении лимита.
- Production – минимальные ответы, содержащие только статус-код и заголовки.
